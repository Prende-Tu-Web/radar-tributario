# Radar Tributario — Blueprint

> Generated by The Architect on 2026-07-14
> Archetype: Marketing Site (primary) + Content Platform (secondary — SEO/lead engine)

---

## 1. Project Overview

### Vision
Radar Tributario (RT) es el rediseño del sitio de marca personal de una experta tributaria, que pasa de ser un blog vitrina sin contenido a un sitio de **generación de leads** para un catálogo real de servicios tributarios, contables y de RRHH, ofrecidos de forma independiente — no a través del empleador actual. El catálogo se organiza en **3 pilares** (Tributario, Contable, RRHH) con 24 servicios específicos entre los tres, más **3 combos de entrada** (ofertas de bajo compromiso para calificar leads más rápido: Diagnóstico, RRHH mensual, Renta anticipada). El sitio combina páginas de servicio con SEO específico por cada una (silo por pilar), un blog que actúa como motor de adquisición (citando fuentes legales chilenas: SII, Banco Central, TGR, Diario Oficial) y múltiples canales de contacto (formulario, WhatsApp, Calendly) para calificar y capturar leads. Ningún precio se publica: todo el catálogo funciona a cotización — los combos aceleran ese primer contacto (calculadora, reserva estacional) pero tampoco muestran precio fijo. No es un e-commerce ni un SaaS: no hay pagos, cuentas de usuario ni checkout. Un CRM a medida para gestionar esos leads se construye en un proyecto aparte y está fuera de este alcance — este sitio solo debe capturar y enviar el lead de forma que ese CRM pueda enchufarse después sin reescribir frontend.

### Goals
- Posicionar a RT (no al empleador) como la marca detrás de los servicios tributarios, contables y de RRHH.
- Convertir tráfico de blog/SEO en leads calificados por servicio y por pilar.
- Página de servicio individual indexable y optimizada para cada una de las 24 líneas de negocio (agrupadas por pilar como silo SEO), incluso si el catálogo cambia — agregar/quitar un servicio no debe requerir tocar código.
- 3 combos de entrada como landing de conversión rápida, independientes del catálogo regular.
- Dejar el sitio listo técnicamente para SEO y ads (GTM/GA4/Meta Pixel) desde el día 1.
- Quedar preparado en local, pulido, antes de publicar — el lanzamiento público es una decisión posterior del usuario.

### Success Metrics
- Leads capturados por formulario/WhatsApp/Calendly, segmentados por servicio de origen.
- Tráfico orgánico a posts de blog y tasa de paso blog → página de servicio → lead.
- Lighthouse ≥ 90 en Performance, SEO y Accessibility en todas las páginas clave.

## 2. Tech Stack

| Layer | Technology | Why |
|-------|-----------|-----|
| Framework | Astro 7 (`output: 'static'` con `export const prerender = false` solo en `/api/lead`) | Ya definido por el usuario; el sitio es 95% contenido estático, solo el endpoint de leads necesita ejecución en servidor. **Nota:** el blueprint original asumía Astro 5 y `output: 'hybrid'` — ese modo se eliminó en Astro 5.0 (fusionado con `static`), y Step 1 instaló Astro 7. Se usa el patrón vigente (`static` + opt-out por página) |
| Language | TypeScript (strict) | Seguridad de tipos en queries a Sanity y en el payload del formulario de leads |
| Styling | Tailwind CSS v4 (`@tailwindcss/vite`) | Setup nativo de Astro, tokens de diseño vía `@theme` en CSS, sin config JS legado |
| Islands interactivos | `@astrojs/react` | Solo para `LeadForm`, `FAQAccordion` y `WhatsAppFloatingButton` — el resto es HTML estático de Astro |
| CMS | Sanity (ya definido — proyecto existente) | Fuente de verdad de contenido; cada servicio y post es un documento editable sin tocar código |
| Cliente Sanity | `@sanity/client` + `@sanity/image-url` + `astro-portabletext` | Queries GROQ tipadas, transformación de imágenes, render de Portable Text incluyendo bloques custom de citas legales |
| Validación | Zod | Valida el payload del formulario de leads en el API route antes de procesar |
| Email transaccional | Resend | Notifica por email cada lead nuevo a la bandeja de RT |
| Integración futura | Webhook genérico (`LEAD_WEBHOOK_URL`) | Punto de enganche para el CRM a medida (proyecto aparte) sin acoplar este sitio a su implementación |
| Analítica/Ads | Google Tag Manager → GA4 + Meta Pixel | El sitio existe para captar leads vía ads; necesitan conversion tracking desde el lanzamiento, no solo analítica |
| Hosting | Vercel | Deploy hooks de Sanity para rebuild al publicar contenido, adapter oficial de Astro, preview deploys |
| Package Manager | pnpm | Estándar del arquetipo, instalación rápida |
| Testing E2E | Playwright | Smoke test de rutas, formulario y enlaces antes de cada release |

## 4. Data Model

El "modelo de datos" de este proyecto vive principalmente en **Sanity** (CMS headless), no en una base de datos relacional propia. El único payload transaccional del lado de la app es el lead enviado desde los formularios.

**`service`** (documento) — 24 documentos iniciales, ver catálogo completo al final de esta sección
| Field | Type | Notes |
|-------|------|-------|
| title | string | Nombre del servicio (ej. "Declaración F29") |
| slug | slug | Genera la ruta `/servicios/[pillar]/[slug]` |
| pillar | string (enum: `tributario` \| `contable` \| `rrhh`) | Define el silo/segmento de la ruta y la columna del mega-menu en el Header |
| icon | string | Nombre de ícono (Lucide/Heroicons) |
| shortDescription | text | Usado en cards (Home, listado del pilar) |
| heroDescription | text | Copy del hero de la página del servicio |
| body | array (Portable Text) | Contenido largo de la página |
| variants | array of { title, description, idealFor } (opcional) | Solo en servicios que agrupan tiers/vías (ej. `contabilidad-mensual` con 3 tiers por tamaño de empresa, `constitucion-empresa` con vía digital/notarial). Renderizado por `ServiceVariantsTable.astro` como tabla comparativa — **nunca incluye precio**, solo alcance/a quién le sirve |

```ts
type LeadPayload = {
  name: string;
  email: string;
  phone: string;
  rut?: string;
```

## 5. API Design

### Routes Overview
| Method | Path | Description | Auth |
|--------|------|-------------|------|
| POST | `/api/lead` | Recibe, valida y procesa un lead (email + webhook opcional) | No (público, protegido por honeypot) |
| GET | `/robots.txt` | Generado dinámicamente, referencia el sitemap | No |

### `/api/lead` — Detalle
1. Parsear body JSON, validar con `leadSchema` (Zod). Si falla → `400` con `{ errors: { field: message } }`.
2. Si `honeypot` no está vacío → responder `200 { success: true }` igualmente (no delatar al bot) pero **no** enviar email ni webhook.
3. Enviar email vía Resend a `contactEmail` (desde `siteSettings` o env var) con los datos del lead y el servicio de origen.
4. Si `LEAD_WEBHOOK_URL` está definido, hacer `POST` fire-and-forget (timeout 3s, no bloquea la respuesta al usuario) con el mismo payload — este es el punto de enganche del futuro CRM.
5. Responder `200 { success: true }` en éxito, `502 { success: false }` solo si Resend falla (el webhook nunca debe bloquear ni fallar la respuesta al usuario).

### State Management
Todo el contenido se resuelve en build-time (SSG) vía queries GROQ a Sanity — no hay estado global de cliente. Revalidación de contenido: Sanity dispara un **Deploy Hook** de Vercel al publicar, lo que fuerza un rebuild completo — no se necesita ISR ni ningún caché intermedio dado el volumen de contenido.

## 7. Design System

> **Historial:** la primera iteración (`/deep-research` + `/ui-ux-pro-max`) reemplazó el azul/Poppins/Open Sans default por una dirección "Editorial Authority" en stone+dorado — ver Sección 16, decisión revertida. **La stakeholder de negocio (no técnica) pidió mantener la paleta real del sitio en vivo** (`radartributario.cl`), extraída directamente de sus CSS custom properties (`--rt-navy/red/teal/warn/bg/border/text/muted`). Se aplicó **solo el cambio de color** — tipografía (Newsreader/Inter), layout, componentes y estructura quedan exactamente igual que en la iteración anterior.

### Colores (paleta real de radartributario.cl)
| Role | Hex | Usage |
|------|-----|-------|
| Primary | `#0F2547` | Navy real de marca — headers, nav activo, texto de alto peso, foco de inputs (así lo usa el sitio real: `.rt-input:focus{border-color:var(--rt-navy)}`) |
| Secondary | `#1E3A5F` | Navy más claro (usado como separador en el sitio real) — **también el color de texto/hover para labels, links y estados hover en toda la UI**, no el accent |
| CTA / Accent | `#22E0C4` | Teal real de marca — reservado **solo como fondo** de botones/badges, nunca como color de texto sobre fondo claro (ver nota de contraste abajo) |
| On Accent | `#0F2547` | Color de texto sobre fondo Accent — el propio sitio pinta texto navy sobre teal (`.rt-tag-teal`), nunca blanco |
| Background | `#F5F4F0` | Fondo general (light mode) |
| Surface | `#FFFFFF` | Cards, paneles |
| Text | `#1E293B` | Texto principal |
| Muted | `#64748B` | Texto secundario, bordes |
| Destructive | `#C0392B` | Rojo real del sitio (`.rt-tag-red`) — errores de formulario |
| Success | `#16A34A` | Sin equivalente real definido — se mantiene el semántico estándar |

> **Nota de contraste (real, no teórica):** `#22E0C4` es un teal claro — como texto sobre `Background`/`Surface` claros falla WCAG AA (~1.4:1). El sitio real ya resuelve esto usando Accent **solo como fondo** con texto navy encima (`On Accent`), nunca como color de texto directo. Todo el sitio sigue esta regla: `text-accent` no se usa en ningún componente — los labels/links/estados-hover que antes usaban el accent dorado (que sí tenía contraste suficiente por ser oscuro) ahora usan `Secondary` (navy claro), que sí pasa contraste sobre fondo claro.

**Sin dark mode, a propósito.** El sitio real no tiene soporte de `prefers-color-scheme` (verificado — 0 coincidencias en su CSS). Se intentó una primera versión con `Primary`/`Surface` invertidos en dark mode (mismo criterio de iteraciones anteriores), pero **causó un bug real**: como estos tokens ahora se usan como colores de marca para fondos sólidos (Header navy, botones navy), no solo como texto, invertirlos en dark mode hacía que el Header se viera blanco y el Hero azul oscuro en navegadores con tema oscuro del sistema — exactamente al revés de lo esperado. Se eliminó el bloque `@media (prefers-color-scheme: dark)` por completo; el sitio se ve igual sin importar el tema del visitante, igual que el original. Implementado como tokens semánticos en `@theme` (Tailwind v4), nunca hex hardcodeado en componentes.

> **Logo:** descargado desde `radartributario.cl/logo.png` a `public/logo.png` (ícono de velocímetro + wordmark "RADAR TRIBUTARIO"). El wordmark "RADAR" del archivo real está diseñado para fondo oscuro (casi invisible sobre blanco) — no se insertó en el Header (fondo claro) para no romper la legibilidad. El Header sigue usando el wordmark tipográfico propio (Newsreader) que ya hereda el navy real automáticamente. Pendiente definir con el usuario dónde usar el archivo de logo real (candidatos: Footer o CTASection, que sí tienen fondo oscuro).

### Typography
| Role | Font | Size | Weight |
|------|------|------|--------|
| Headings | Newsreader | Escala 14/16/20/24/32/40/48 | 500–700 (cursiva opcional en citas destacadas del blog) |
| Body | Inter | 16px base, line-height 1.6 | 400 (500 para labels) |
| Fuente import | `@import url('.../Inter:wght@300;400;500;600;700&family=Newsreader:wght@400;500;600;700')` self-hosted en `public/fonts/` con `font-display: swap` | — | — |

Pairing "Editorial Authority": Newsreader es una serif diseñada específicamente para lectura larga editorial — encaja con el blog/noticias como motor de adquisición (posts extensos, citas legales) y transmite autoridad periodística en vez de "look de SaaS". Inter como sans neutro para UI, formularios y navegación mantiene la interfaz legible y moderna sin competir con la personalidad de los headings. Reemplaza Poppins + Open Sans (dos geométricas) — identificado en la investigación previa como pairing repetido en sitios genéricos/IA.

### Spacing & Layout
- Spacing scale: base 4px — 4, 8, 12, 16, 24, 32, 48, 64, 96
- Border radius: 8px default, 12px cards, full para avatares/badges
- Líneas finas (`border` 1px, color `Muted` al 20-30% opacidad) como elemento de diseño intencional para separar secciones — refuerza el tono editorial sin depender solo de sombras
- Max content width: 1280px (`max-w-7xl`)
- Breakpoints: 375 / 768 / 1024 / 1440 (mobile-first)

### Component Style
Patrón **Trust & Authority**: cards con credenciales/certificaciones visibles, badges de confianza, métricas si existen (años de experiencia, casos resueltos). Sombras suaves (`shadow-sm`/`shadow-md`), radios moderados (no brutalist, no glassmorphism), transiciones 150–300ms en hover/focus, íconos SVG (Lucide) — nunca emojis como íconos estructurales. Un único CTA primario visible por sección (color `CTA`); acciones secundarias en `outline`/`ghost`.

## 13. Testing Strategy

### Unit Tests
No es prioritario para este proyecto (sitio mayormente estático) — opcional: test unitario de `leadSchema` (Zod) con casos válidos/inválidos si se quiere blindar el endpoint.

### Integration Tests
Test del endpoint `/api/lead`: payload válido → 200, payload inválido → 400 con errores por campo, honeypot lleno → 200 pero sin envío de email (mockear Resend).

### E2E Tests
Playwright cubriendo: todas las rutas cargan (Home, Servicios overview + pilares + servicios dinámicos, Combos index + los 3 combos, Noticias index + 1 post + 1 categoría, Autor, Sobre Nosotros, Contacto), envío de `LeadForm` happy path desde Servicio y desde Combo, envío del `HeadcountCalculator`, navegación del Header/mega-menu/Footer, verificación de que WhatsApp button abre `wa.me` con el número correcto. Ejecutar antes de cada deploy a producción.

**Implementado en `tests/e2e/` (44 tests, chromium + mobile). El QA real encontró y corrigió 3 bugs que habrían llegado a producción:**
1. **`/noticias` con paginación en un archivo `index.astro`** — Astro requiere el archivo `[...page].astro` (rest param) para que `paginate()` genere `/noticias`, `/noticias/2`, etc. desde el mismo componente; `index.astro` + `paginate()` fallaba en build con `Cannot read properties of undefined (reading 'data')`.
2. **`LeadForm` llamaba `fetch('/api/lead')` sin barra final** — con `trailingSlash: 'always'` (Sección 6), esto caía en 404 antes de llegar al endpoint. El formulario habría estado completamente roto en producción, silenciosamente.
3. **Condición de carrera de hidratación en `LeadForm`** — con `client:visible`, Playwright (y potencialmente un usuario escribiendo muy rápido) podía llenar el primer campo antes de que React terminara de hidratar; al hidratar, el componente pisaba el valor tipeado con su estado inicial vacío. Se cambió a `client:load` en las 4 instancias del formulario (Home, Contacto, Servicio, Combo) — es la acción de conversión principal del sitio, justifica hidratar sin esperar a que entre en el viewport.

```markdown
## Architecture

### Directory Structure
- `src/pages/` — Rutas: Home, Servicios (overview + [pillar] + [pillar]/[slug]), Combos (index + [slug]), Noticias (index + [slug] + categoria/[slug]), Autor ([slug]), Sobre Nosotros, Contacto, `api/lead.ts`

- `sanity/schemas/` — Referencia a los schemas del Studio existente (service, combo, post, author, siteSettings, legalSource)

### Data Flow
Todo el contenido (servicios, posts, configuración del sitio) se resuelve en build-time vía queries GROQ a Sanity — no hay base de datos propia. El único flujo server-side es `POST /api/lead`: valida con Zod → envía email vía Resend → opcionalmente reenvía a `LEAD_WEBHOOK_URL` (futuro CRM, fire-and-forget). Sanity dispara un Deploy Hook de Vercel al publicar contenido para forzar rebuild.

### Key Patterns
```

## 16. Reglas No Negociables

1. **No autenticación ni cuentas de usuario.** Es un sitio público de generación de leads, no un SaaS.
2. **El CRM es un proyecto aparte.** Este sitio solo debe exponer un webhook opcional y configurable (`LEAD_WEBHOOK_URL`) que funcione en no-op si está vacío — nunca construir lógica de CRM dentro de este repo.
3. **Sanity es la fuente de verdad de contenido.** Ningún servicio, post o dato editorial se hardcodea en el código; y antes de crear schemas nuevos, se audita lo que ya existe en el dataset actual (este sitio se acopla a un Sanity existente, no parte de cero).
4. **No deploy a producción sin instrucción explícita del usuario.** El build y `pnpm preview` en local son el criterio de "listo", no un dominio público.
5. **TypeScript strict, sin `any`.** Los tipos de Sanity y el payload de leads deben estar tipados de punta a punta.
6. **Formulario de leads: honeypot + validación Zod server-side siempre**, sin CAPTCHA (prioriza UX/conversión).
7. **Un único CTA primario visible por sección/página** — el color de acento (`#A16207`) está reservado exclusivamente para llamadas a la acción, nunca como color decorativo.
8. **Mobile-first.** Diseñar y probar primero en 375px — la mayoría del tráfico de blog y ads llega desde mobile.
9. **Todo color/tipografía vía tokens semánticos** en `@theme` de Tailwind v4 — nunca hex hardcodeado dentro de un componente, para que el swap de paleta/logo definitivo (pendiente) sea un cambio de un solo archivo.
10. **Ningún precio público en el sitio.** Ni en `service`, ni en `service.variants`, ni en `combo` — todo el catálogo funciona a cotización vía `LeadForm`. El `HeadcountCalculator` del combo RRHH mensual usa el N° de trabajadores solo para precargar el lead, nunca para mostrar una tarifa.
